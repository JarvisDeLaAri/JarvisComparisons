⏰ Cron Job Settings

Cron jobs run at precise times and can execute in isolated sessions. Perfect for exact scheduling like "9:00 AM every Monday" or one-shot reminders.

Schedule Options

schedule.kind Cron
What it does: Tells cron HOW to schedule - run once, repeat forever, or on a schedule.
Values: "at", "every", "cron"
schedule.at Cron
What it does: Sets the exact date/time for one-shot jobs. Like an alarm clock.
Example: "2026-02-01T16:00:00Z"
Format: ISO 8601 (timezone optional, UTC if omitted)
schedule.everyMs Cron
What it does: Repeat every X milliseconds. Good for "check every 5 minutes".
Example: 300000 = 5 minutes
Type: number (milliseconds)
schedule.expr Cron
What it does: Standard cron expression for recurring jobs. "At 7 AM every day" = "0 7 * * *"
Format: "minute hour day month weekday"
Example: "0 7 * * *" (7 AM daily)
schedule.tz Cron
What it does: Which timezone to use for the schedule. No more "why did it run at 5 AM?!"
Example: "America/New_York", "Asia/Jerusalem"
Type: IANA timezone string
schedule.staggerMs Cron
What it does: Adds random delay (0-5 min by default) to prevent all jobs running at once.
Example: 30000 = 30 seconds
Default: 0-5min random for top-of-hour jobs

Session & Execution

sessionTarget Cron
What it does: Where the job runs - main session (with your chat history) or isolated (fresh start).
"main" = your normal chat
"isolated" = fresh session each time
"session:custom-id" = named persistent session
wakeMode Cron
What it does: When to actually run - immediately or wait for next heartbeat.
"now" = run right away (default)
"next-heartbeat" = wait for next check-in
agentId Cron
What it does: Run this job using a specific agent. Useful in multi-agent setups.
Example: "ops", "main"
If omitted, uses default agent

Payload (What Runs)

🤔 System Event vs AI Turn - What's the Difference?

🖥️ System Event = A Nudge to Your Chat

Simple explanation: Think of it like a note dropped into your current conversation. The AI sees it during the NEXT time it talks to you (next heartbeat or next message). It has full context of everything you've discussed.
(1) You give prompt to your session. this prompt is injected to your agent session (say your whatsapp session with your main agent with all your conversation history). (2) BUT you dont see the injection. IF agent decides to react to you with a message, (3) you see reply.
• Runs in main session (your actual chat)
• AI sees your full conversation history
• Good for: reminders, nudges, things you want AI to remember
• Example: "Reminder: check calendar" → shows up in your chat

🤖 AI Turn = A Background Worker

Simple explanation: Think of it like hiring a helper who does a task independently, then reports back. They DON'T see your chat history - they start fresh each time. They can use a different AI model and send results anywhere.
(1) You give prompt to NEW agent session. this prompt is creating new ISOLATED session with your agent (with all his files, but no conversation history). (2) BUT you AND agent dont see the injection . IF you use announce it sends you a message to channel (say whatsapp), (3) you see reply BUT NOT AGENT.
• Runs in isolated session (fresh, no history)
• AI starts with blank context each run
• Good for: reports, summaries, anything noisy or independent
• Example: "Morning briefing" → AI generates report → sends to WhatsApp
📋 Quick Decision Guide:
  • Want the AI to DO something in YOUR chat? → System Event
  • Want a background task that reports elsewhere? → AI Turn
  • Need full conversation context? → System Event
  • Want to use a different AI model? → AI Turn
payload.kind Cron
What it does: Tells cron WHAT to do - send a system event or run an AI turn.
"systemEvent" = main session event
"agentTurn" = isolated AI run
payload.text / payload.message Cron
What it does: The actual message or prompt that gets executed.
payload.text = for systemEvent
payload.message = for agentTurn
payload.model Cron
What it does: Use a different AI model for this job.
Example: "openai-codex/gpt-5.3-codex", "minimax-m2.5:cloud"
Uses main model if omitted
payload.thinking Cron
What it does: How much "thinking" the AI should do. More thinking = better results but more expensive.
Values: "off", "minimal", "low", "medium", "high", "xhigh"
payload.timeoutSeconds Cron
What it does: Kill the job if it takes too long. Prevents runaway AI from costing too much.
Example: 300 = 5 minutes
Default: no timeout (0)
payload.lightContext Cron
What it does: Skip loading workspace files. Load NOTHING for cron.
true  = no context loading
false = full workspace (default)

Delivery (Where Results Go)

delivery.mode Cron
What it does: How to send results - just announce, post to webhook, or silent.
"announce" = send to channel (default)
"webhook" = POST to URL
"none" = silent, no delivery
delivery.channel Cron
What it does: Which chat platform to send the result to.
Options: "whatsapp", "telegram", "discord", "slack", "last"
delivery.to Cron
What it does: Specific recipient - phone number, chat ID, channel name.
WhatsApp: "+15551234567"
Telegram: "-1001234567890:topic:123"
Discord: "channel:C123456"
delivery.bestEffort Cron
What it does: If delivery fails, don't worry about it. Job still counts as success.
true = don't fail on delivery error
false = fail if can't deliver (default)

Other Options

name Cron
What it does: Human-readable name so you know what the job does.
Example: "Morning briefing"
Shows in cron list and logs
enabled Cron
What it does: Turn the job on or off without deleting it.
true = runs on schedule
false = paused
deleteAfterRun Cron
What it does: One-shot jobs disappear after running. Like a self-destructing note.
true = delete after success (default for "at" jobs)
false = keep running forever

💓 Heartbeat Settings

Heartbeats run periodically in the main session, allowing the AI to check on things and surface important updates. Perfect for batched monitoring like "check inbox + calendar + notifications".

Timing

every Heartbeat
What it does: How often to check in. Default is 30 minutes - like a regular check-in.
Example: "30m", "1h", "5m"
Default: "30m" | Use "0m" to disable
activeHours.start Heartbeat
What it does: Don't bother me before this time. Morning people set 6 AM, night owls set noon.
Format: "HH:MM"
Example: "09:00"
activeHours.end Heartbeat
What it does: Stop checking after this time. Sleep hours = no interruptions.
Format: "HH:MM"
Example: "22:00" (10 PM)
Use "24:00" for end of day
activeHours.timezone Heartbeat
What it does: Which timezone to use for active hours. Keeps your schedule consistent.
Example: "America/New_York", "Asia/Jerusalem"
Default: your userTimezone or host timezone

Session & Context

session Heartbeat
What it does: Which session to run the heartbeat in. Default main, but can target specific channels. (non-isolated only)
"main" = default session
Or specific session key from openclaw sessions
isolatedSession Heartbeat
What it does: Start fresh each time without your chat history. Way cheaper on tokens!
true = fresh session, ~2-5K tokens
false = full history, ~100K+ tokens
lightContext Heartbeat
What it does: Only load HEARTBEAT.md, skip all other workspace files. Faster and cheaper.
true = only HEARTBEAT.md
false = full workspace (default)

Delivery

target Heartbeat
What it does: Where to send heartbeat alerts. "last" = wherever you last talked. or choose channel ("whatsapp", "telegram").
"none" = don't send anywhere (default)
"last" = last used channel
"whatsapp", "telegram", etc. = send to that channel
to Heartbeat
What it does: Specific phone number or chat ID to send to. Overrides the target default.
Example: "+15551234567"
Or Telegram: "12345678:topic:42"
accountId Heartbeat
What it does: For multi-account channels (like Telegram bots). Send from a specific account.
Example: "ops-bot"
Matches channel accounts config
directPolicy Heartbeat
What it does: Allow or block direct messages (DMs). Useful for group chats.
"allow" = allow DM delivery (default)
"block" = suppress DM delivery

Model & Reasoning

model Heartbeat
What it does: Use a different (usually cheaper) model for heartbeat checks.
Example: "ollama/llama3.2:1b"
Defaults to main session model
includeReasoning Heartbeat
What it does: Also show the AI's "thinking" process. Like showing your work in math class.
true = send separate Reasoning: message
false = answer only (default)

Prompt & Response

📬 Heartbeat Response - What Do You See?

⚙️ The ackMaxChars Threshold

Default: 30 characters

If content after HEARTBEAT_OK is ≤ 30 chars → sends NOTHING (not worth waking you up)
If content is > 30 chars → sends that content to you
If no HEARTBEAT_OK token at all → sends FULL response

ackMaxChars can be set.

⚙️ The showOk Parameter

Default: false

if true sends HEARTBEAT_OK

⚙️ how to always get response

use ackMaxChars = 0 and showOk = true

🔍 How It Decides What to Send (default settings)

Simple explanation: The AI responds with "HEARTBEAT_OK" when nothing needs attention. If there's something important, it adds content after that token.
AI says: "HEARTBEAT_OK" (nothing wrong) → you see NOTHING (skipped)
AI says: "HEARTBEAT_OK check your inbox" → you see: "check your inbox" (content after the token, IF > 30 chars)
AI says: "Checked inbox, found 3 urgent emails from client" → you see FULL response (no HEARTBEAT_OK token)
prompt Heartbeat
What it does: Custom instructions for each heartbeat. Overrides the default "read HEARTBEAT.md" prompt.
Default: "Read HEARTBEAT.md if it exists..."
Custom prompt sent verbatim
ackMaxChars Heartbeat
What it does: Threshold to decide if response should be delivered. If AI says "HEARTBEAT_OK" plus short content, skip it.
Default: 30 characters
If response after HEARTBEAT_OK is ≤ this → sends nothing
If > this → sends content after HEARTBEAT_OK
suppressToolErrorWarnings Heartbeat
What it does: Don't show tool errors in heartbeat responses. Cleaner, less noise.
true = suppress warnings
false = show all errors (default)

Visibility Controls (Per Channel)

showOk Heartbeat
What it does: Show the "HEARTBEAT_OK" message when nothing is wrong.
true = show OK messages
false = hide OK messages (default)
showAlerts Heartbeat
What it does: Show actual alerts when something needs attention.
true = show alerts (default)
false = suppress alerts
useIndicator Heartbeat
What it does: Show a typing indicator or status in the UI.
true = show indicator (default)
false = no indicator

⚙️ Global Cron Configuration

These settings go in the gateway config under the cron section. They control overall cron behavior, not individual jobs.

cron.enabled Global
What it does: Turn all cron jobs on or off completely.
true = enabled (default)
false = disabled
cron.store Global
Where it saves jobs: Custom location for the jobs.json file.
Default: "~/.openclaw/cron/jobs.json"
cron.maxConcurrentRuns Global
What it does: How many cron jobs can run at the same time. More = faster but more resources.
Default: 1
Increase for high-volume setups
cron.sessionRetention Global
What it does: How long to keep isolated cron session history before cleanup.
Default: "24h"
Set false to disable
cron.runLog.maxBytes Global
What it does: Maximum size of the run log file before it gets trimmed.
Default: "2mb"
Example: "10mb"
cron.runLog.keepLines Global
What it does: When trimming logs, keep only the newest N lines.
Default: 2000 lines

Retry Configuration

cron.retry.maxAttempts Global
What it does: How many times to retry a failing one-shot job before giving up.
Default: 3
cron.retry.backoffMs Global
What it does: Wait times between retries. Starts short, gets longer (exponential backoff).
Default: [60000, 120000, 300000]
1 min → 2 min → 5 min
cron.retry.retryOn Global
What it does: Which errors should trigger a retry. Network issues = yes, bad config = no.
Options: "rate_limit", "overloaded", "network", "server_error"

🎯 When to Use What

⏰ Use Cron When...

  • You need exact timing ("9:00 AM sharp")
  • It's a one-shot reminder ("in 20 minutes")
  • Task should run independently of main chat
  • You want to use a different AI model
  • Task is noisy/frequent - keep it out of main history
  • Results should go to a specific channel

💓 Use Heartbeat When...

  • Regular awareness checks (inbox, calendar)
  • Can batch multiple checks together
  • Needs full chat context to decide
  • Timing can drift slightly (~30 min is fine)
  • Want smart suppression if nothing to report
  • Occasional check-in without scheduling

📝 Example Configs

Cron Job - Daily Morning Briefing

# Run at 7 AM every day, send to WhatsApp { "name": "Morning briefing", "schedule": { "kind": "cron", "expr": "0 7 * * *", "tz": "Asia/Jerusalem" }, "sessionTarget": "isolated", "payload": { "kind": "agentTurn", "message": "Generate today's briefing: weather, calendar, emails.", "model": "opus" }, "delivery": { "mode": "announce", "channel": "whatsapp", "to": "+972542634114" } }

Heartbeat - Periodic Awareness

# Check every 30 minutes during daytime, deliver to last used channel { "heartbeat": { "every": "30m", "target": "last", "lightContext": true, "activeHours": { "start": "08:00", "end": "23:00" } } }

One-Shot Reminder

# Remind me in 20 minutes via main session { "name": "Meeting reminder", "schedule": { "kind": "at", "at": "2026-03-22T18:30:00Z" }, "sessionTarget": "main", "wakeMode": "now", "payload": { "kind": "systemEvent", "text": "Reminder: Standup meeting in 10 minutes!" }, "deleteAfterRun": true }
