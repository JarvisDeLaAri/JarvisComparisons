🤖 OpenClaw Personalization

Messages, Reasoning & Thinking — Configure how your AI assistant thinks, responds, and interacts

OpenClaw provides fine-grained control over message handling, reasoning visibility, and thinking effort. This guide covers all three personalization layers with config paths and practical examples.

💬 Messages Configuration

Control how messages are processed, queued, acknowledged, and delivered across different channels.

messages.messagePrefix config

Prefix text prepended to inbound user messages before they reach the agent. Use for channel context markers.

advanced
messages.responsePrefix config

Prefix text prepended to outbound assistant replies. Use for branding tags or context markers.

advanced
messages.groupChat config

Group-message handling controls: mention triggers and history window sizing.

advanced
messages.queue config

Inbound message queue strategy. Controls how bursts of messages are handled before processing.

advanced
messages.queue.mode config

Queue behavior: steer, followup, collect, steer-backlog, steer+backlog, queue, or interrupt.

advanced
messages.queue.debounceMs config

Global queue debounce window in milliseconds. Higher values coalesce bursts, lower values reduce latency.

performance
messages.ackReaction config

Emoji reaction used to acknowledge inbound messages. Empty disables.

advanced
messages.statusReactions config

Lifecycle status reactions (queued → thinking → tool → done/error). Currently for Telegram.

advanced
messages.inbound.debounceMs config

Debounce window for rapid inbound messages from the same sender (0 to disable).

performance
messages.tts config

Text-to-speech policy for reading agent replies aloud on supported voice surfaces.

media
messages.suppressToolErrors config

When true, suppress ⚠️ tool-error warnings from being shown to the user. Default: false.

advanced
messages.queue.drop config

Drop strategy when queue cap exceeded: old, new, or summarize.

advanced

Queue Mode Reference

Mode Behavior
steer Process latest, drop older queued items from same sender
followup Append to existing turn context (single response)
collect Batch multiple messages before responding
queue Process each message sequentially
interrupt Stop current turn, start new one with latest message
Quick Tip: Start with steer (default) for most channels. Use interrupt only when you want aggressive new-message-priority behavior.

🧠 Reasoning Configuration

Control whether the model's internal reasoning/thinking is visible to users.

off

Completely hide reasoning. Users only see the final response.

on

Show reasoning content after the response completes. Most common for production.

stream

Stream reasoning in real-time as the model thinks. Great for debugging.

Configuration Levels

Session Level (per-session)
Set via reasoningLevel in session store
Agent Default
agents.list[].reasoningDefault
Runtime Override
Use /reasoning on|off|stream in chat
Channel Consideration: WhatsApp and Signal don't support streaming reasoning — users only see the final answer. Streaming works best in TUI, Web UI, and Discord.

💭 Thinking Configuration

Control how much computational effort the model spends on reasoning (affects quality vs. latency).

off

No extended thinking. Fastest response, minimal reasoning effort.

minimal

Minimal thinking effort. Quick responses with basic reasoning.

low

Low thinking effort. Suitable for simple tasks.

medium

Balanced thinking. Default for most use cases.

high

High thinking effort. For complex reasoning tasks.

xhigh

Maximum thinking. Deep reasoning, slower but thorough.

Configuration Levels

Session Level (per-session)
Set via thinkingLevel in session store
Agent Default
agents.list[].thinkingDefault
Runtime Override
Use /think minimal|low|medium|high|xhigh in chat
Provider Default
Some models have built-in defaults (e.g., Claude 4.6 → adaptive)

Key Differences

Aspect Reasoning Thinking
What it controls Visibility of reasoning to users Computational effort / reasoning depth
Options on, off, stream off, minimal, low, medium, high, xhigh, adaptive
Affects response time No (just visibility) Yes (higher thinking = slower)
Per-channel relevance Yes (some channels don't support streaming) No (model-side configuration)

⚙️ Practical Examples

Example 1: Fast & Minimal WhatsApp Bot

// For fast WhatsApp responses, disable extended thinking { "thinkingLevel": "low", "reasoningLevel": "off", "messages": { "queue": { "mode": "steer", "debounceMs": 500 } } }

Example 2: Debug Mode with Full Visibility

// For development/debugging - see everything { "thinkingLevel": "high", "reasoningLevel": "stream", "messages": { "statusReactions": { "enabled": true } } }

Example 3: Group Chat Optimization

// Optimized for busy group chats { "messages": { "groupChat": { "mentionPatterns": ["@jarvis", "!jarvis"], "historyLimit": 20 }, "queue": { "mode": "interrupt", "cap": 5, "drop": "old" } } }
