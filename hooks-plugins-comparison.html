🪝 Hooks vs Plugins

When to use which — the practical guide for extending OpenClaw

Built-in

🪝 Hooks

Lightweight scripts that run inside the Gateway when specific events fire

Setup Zero config
Run location Inside Gateway
Trigger type Event-based
Examples /new, /reset, /stop, lifecycle
Complexity Low

✅ Pros

  • Enabled by default (session-memory, command-logger, boot-md)
  • No external dependencies
  • Super fast — runs in-process
  • Great for automation on session events
  • TypeScript functions, simple

❌ Cons

  • Only responds to internal events
  • Can't add new channels or model providers
  • Limited to Gateway lifecycle
  • Not for external integrations
Extension

🔌 Plugins

Full npm packages that extend OpenClaw with new capabilities

Setup Install + config
Run location Separate process
Trigger type On-demand
Examples Matrix, Teams, Voice, web-search
Complexity Medium-High

✅ Pros

  • Can add entirely new channels (WhatsApp, Discord, Matrix)
  • New model providers
  • Tools, speech, image generation
  • Published on npm — easy install
  • Can bundle hooks inside them

❌ Cons

  • Need to install & configure
  • Restart gateway after install
  • More moving parts
  • Security: trust the npm package

🪝 How Hooks Work

1
List
openclaw hooks list
2
Enable
openclaw hooks enable session-memory
3
Trigger
Event fires (/new, /reset, etc)
4
Run
Hook executes in Gateway
💡 Pro tip: Plugins can bundle hooks inside them! So you can get both — new channels PLUS event automation in one package.

🎯 The Verdict

Use Hooks for internal automation — session logging, memory saves, audit trails, anything that reacts to your own commands.

Use Plugins when you need to connect to the outside world — new chat platforms, external APIs, model providers, or specialized tools.

Start with hooks for quick wins. Graduate to plugins when you need to scale or integrate.
